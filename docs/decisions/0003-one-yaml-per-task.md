# 0003. One YAML file per task; machine state lives elsewhere

- Status: accepted
- Date: 2026-04-19
- Deciders: @billdenney

## Context

We needed a format for the todo list that is (a) easy for humans to
write by hand, (b) easy to diff in version control, and (c) safe to
update from the runner without clobbering hand edits.

## Decision

Each task is its own YAML file under a todo directory (default
`./todo/`). The file is authored by the user (or scaffolded by
`claude_runner new`) and contains the task definition. Machine-written
state — status, session id, run history, token usage — lives in a
parallel, gitignored directory `.claude_runner/state/<id>.yaml`.

Only `prompt` and `working_dir` are required in the definition file;
every other field falls back to the user's config and the package
defaults (see ADR 0009).

## Alternatives considered

- **Single `todo.yaml` manifest** — rejected: editing one task means
  touching a big file; Git history is harder to read; concurrent edits
  conflict constantly.
- **Definition and state in the same file** — rejected: every token
  update becomes a diff on a user-authored YAML, which guarantees lost
  edits and noisy `git status`.
- **SQLite queue DB** — rejected for v0.1 because it sacrifices the
  "hand-edit and commit your todos" workflow.

## Consequences

- **Positive:** the todo directory is a first-class artifact that can be
  committed, reviewed, and shared; the state directory is strictly
  internal and can be wiped to start fresh.
- **Negative:** reading "the state of a task" requires looking at two
  places. The CLI's `status` subcommand joins them and is the
  recommended view.
- **Follow-ups:** `.gitignore` generated by `claude_runner init`
  excludes the state tree so users don't commit it by accident.
